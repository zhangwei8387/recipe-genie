## 项目概述
- **目标**：构建面向健康饮食者与上班族的 AI 驱动实时交互式食谱推荐平台。
- **目标用户**：🥗 健康饮食者（关注营养）、👔 上班族（时间有限）、兼顾探索型用户。
- **核心场景（优先级）**：
  1. 📦 食材反推食谱（"冰箱里有 X，能做什么？"）
  2. ⏰ 时间限制推荐（"30 分钟能做什么？"）
  3. 💪 营养目标匹配（"高蛋白低脂"）
  4. 🎭 口味/菜系偏好（"想吃川菜"）
  5. 🤔 智能探索（"不知道吃什么"）

## Vibe Coding 原则（首选） 🔥
> 目标：尽量通过 AI 生成、模板化与自动化实现功能，减少重复的人为手动编码工作，但保留必要的人类评审与质量门控。

**实施要点**：
- 文档优先：所有需求先在 `docs/` 完整描述（PRD、用户故事、验收标准）。
- 模板与自动化：使用 `docs/templates/` 的模板生成 prompts、任务与代码脚手架。
- AI 生成代码的质量门：生成的代码必须通过 CI（lint、单元测试）、并经过至少一名人类审查后才能合并。📋
- 生成记录与可审计性：保存用于生成的 prompts/配置与审查注释，建议放置于 `docs/ai-prompts/`（或相似位置）。
- 手写代码情形：当自动化无法覆盖复杂业务逻辑、安全/合规性需求或性能关键路径时，由开发者手写并在 PR 中明确理由。
- 风险控制：对安全、隐私和食品安全相关逻辑应设置更严格的人工审核流程。

**好处**：减少模版样板类工作、加快迭代、便于通过可重复流程扩展功能。

## 快速开始（Vibe 工作流）
1. 在 `docs/requirements/` 完成或更新 PRD/用户故事与验收标准。  
2. 在 `docs/sprints/` 或 `docs/templates/` 使用任务模板创建 Sprint 任务并填好负责人与估时。  
3. 使用 AI 工具（或内部脚手架）生成代码/配置。  
4. 提交 PR：确保 CI 通过、包含测试、并至少一名审查者批准后合并。  
5. 发布与观测：部署后监控 KPI（见 PRD 中的 KPI 定义）。

> 备注：目前仓库以文档为先（`docs/`），避免直接添加大量代码文件。请先在文档中定义需求与验收标准。

## 开发与贡献简要
- **贡献流程**：通过 GitHub 提交 PR；PR 描述应包含变更摘要、关联 issue/任务与验收项。  
- **分支策略**：推荐 `feature/*`、`fix/*` 等命名；提交信息建议使用 Conventional Commits。  
- **质量要求**：所有代码需通过 CI（lint、测试）并经人工审查；AI 生成代码同样适用此规则。
